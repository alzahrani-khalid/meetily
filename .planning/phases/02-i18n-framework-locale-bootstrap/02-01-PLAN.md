<objective>
Add a persistent `bootstrapped` boolean flag to the Phase 1 preferences column set so that Phase 2's first-run detector can mark detection as "done" exactly once per install and never silently re-detect on subsequent launches.

Purpose: D-01..D-04 locked decision — a column, not a re-detect-on-every-launch heuristic — prevents the correctness bug where a deliberate English pick silently flips to Arabic when the user travels to an Arabic-locale machine.

Output:
1. Additive SQLite migration `20260409000000_add_preferences_bootstrapped_flag.sql` adding one INTEGER column defaulting to 0.
2. `UserPreferences` Rust struct with `pub bootstrapped: bool`, `merge()` respecting it, repository.rs hydrating it, `UserPreferencesPatch` accepting it.
3. `PREFS_CACHE` default row seeded with `bootstrapped: false`.
4. TypeScript `UserPreferences` + `UserPreferencesPatch` types extended to include `bootstrapped: boolean` / `bootstrapped?: boolean`.
5. Phase 1's existing T1..T5 Rust tests continue to pass unchanged (verified via `cargo test`).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tauri IPC (frontend JS → Rust command) | Phase 2 adds one new field to the existing `set_user_preferences` command's patch shape. The field is a boolean. |
| SQLite file system boundary | Migration writes an ALTER TABLE. No new write path — flows through Phase 1's `apply_patch_atomic` with its existing invariants. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `bootstrapped` column | accept | The column is a local SQLite boolean. A user with filesystem access can flip it manually; this is the same trust level as any other preference field. No remote attacker has write access (Tauri IPC is process-local, no HTTP surface). |
| T-02-02 | Information Disclosure | `bootstrapped` field over IPC | accept | The value is a non-sensitive boolean. It leaks "has this install ever been launched" — which is already discoverable via `updated_at != 0` on the existing schema. Zero new information surface. |
| T-02-03 | Denial of Service | Migration on startup | mitigate | ALTER TABLE with DEFAULT 0 is an O(n) operation, but `user_preferences` has exactly 1 row. Migration completes in milliseconds. No user-facing delay. Verified by manually running the migration against a seeded DB during Task 2. |
| T-02-04 | Elevation of Privilege | `UserPreferencesPatch.bootstrapped` write | accept | The field has no security-critical semantics — it gates a one-time UI detection. A malicious caller setting `bootstrapped: false` would re-trigger the detector, which is still confined to local preferences. No privilege boundary crossed. |
| T-02-05 | Repudiation | Migration rollback | mitigate | Additive migration is forward-only. If rollback is ever needed, the executor drops the column via a follow-up migration. No rollback required for Phase 2 scope. Phase 1 T1..T5 passing after the migration is the forward-safety proof. |
| T-02-06 | Supply chain | No new dependencies | accept | This plan adds ZERO npm or cargo dependencies. Nothing new to audit. |

**Severity summary:** No HIGH findings. All LOW/accept are justified by the field's local, non-sensitive semantics. Plan is unblocked.
